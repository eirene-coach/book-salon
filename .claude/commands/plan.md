# 기능 계획 → 백로그 티켓 생성

기능 아이디어를 분석하여 구조화된 백로그 티켓들로 분해합니다.
인자로 기능 설명을 받습니다. (예: `/plan 사용자 인증 기능 추가`)

## 보드 찾기
- `.claude/settings.json`의 `mcpServers.vuild.env.VUILD_BOARD_ID`가 있으면 해당 보드를 사용합니다.
- 없으면 CLAUDE.md의 `Board ID`를 확인합니다.
- 둘 다 없으면 `board_list` 결과를 확인합니다 (1개면 자동 선택, 여러 개면 사용자에게 선택 요청).
- 보드가 아예 없으면 "이 프로젝트에 연결된 Dev 보드가 없습니다. `/init-project`로 새 프로젝트/보드를 생성하세요." 안내 후 종료.
- MCP 연결 실패 → `/setup`으로 MCP 설치 안내.

## 사용 가능한 에이전트 (.claude/agents/)

| 에이전트 | 역할 | spawn 시점 |
|---------|------|-----------|
| 팀리드 (`00-team-lead.md`) | 작업 조율, 에이전트 위임, 진행 관리 | 전체 |
| Analyzer (`01-analyzer.md`) | 코드베이스 분석, 영향 범위 파악 | 분석 단계 |
| Feature Planner (`06-feature-planner-agent.md`) | 작업 분해, PR 분할, TDD 워크플로우 | 계획 수립 |

## 실행 순서

### 1. 현재 보드 파악
- `board_get_summary`로 프로젝트 상태를 확인합니다
- `board_list_tickets` (status: "backlog")로 기존 백로그와 중복이 없는지 확인합니다

### 2. 분석 및 계획
- 코드베이스를 분석하여 관련 파일, 기존 패턴, 영향 범위를 파악합니다
- 구체적인 작업 단위로 분해합니다 (각 작업이 1개 PR 크기)
- 각 작업에 제목, 설명, 우선순위, 완료 기준, 의존 관계를 설정합니다
- 각 작업의 **병렬 실행 가능 여부**를 판단합니다:
  - 수정 파일이 겹치지 않고, DB 마이그레이션이 없으면 `[parallel: yes]`
  - 의존성이 있으면 `[parallel: after:#티켓ID]`
  - 같은 모델/파일 수정 시 `[parallel: no]`

### 3. 계획 비판 리뷰
- 분해 결과를 사용자에게 보여주기 **전에**, 계획 자체의 허점을 검토합니다
- 다음 관점에서 스스로 비판적 리뷰를 수행합니다:
  - "시니어 엔지니어라면 이 계획의 허점은 무엇이라고 생각할까?"
  - 누락된 엣지케이스가 있는가?
  - 과소평가된 복잡성이 있는가?
  - 작업 간 의존성 충돌이 있는가?
  - 기존 코드와의 충돌 가능성이 있는가?
- 발견된 허점과 보완안을 계획에 반영한 후 다음 단계로 진행합니다

### 4. 사용자 확인
- 계획 + 리뷰에서 발견된 리스크를 함께 제시합니다
- 형식:
  ```
  ■ 계획 요약
    - [작업 목록]

  ■ 리뷰 발견사항
    - [허점/리스크와 보완안]

  ■ 검증 기준
    - 각 티켓의 완료 판정 명령어
  ```
- 사용자 승인 후 티켓 생성

### 5. 백로그 티켓 생성
- 승인받은 작업들을 `board_create_ticket`로 생성합니다:
  - `board_id`: CLAUDE.md의 Board ID
  - `status`: "backlog"
  - `priority`: 설정한 우선순위
  - `description`: 반드시 아래 두 섹션을 포함:
    - `## 완료 기준` — 무엇이 되면 완료인지 (텍스트)
    - `## 검증 명령어` — 완료를 판정할 실행 가능한 명령어 (예: `bin/rails test test/models/user_test.rb`)
- `board_add_activity`로 계획 기록

### 5. 결과 보고
```
[기능명] 계획 완료:
  생성된 티켓: [N]개 (backlog)
  - [티켓1] (priority: high)
  - [티켓2] (priority: medium)

다음 단계:
  - 바로 작업 시작 → /backlog
  - 현황 확인 → /dashboard
```
